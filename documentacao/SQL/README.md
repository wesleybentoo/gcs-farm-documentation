SQL canônico dos bancos `CONNECTOR_GCS_FARM` (raw) e `GCS_FARM` (master). **Em sync com o banco vivo (0 drift — reconferido 03/07 rodando o `SETUP_FULL.sql` num banco novo `*_TEST`: mesmas contagens e mesmos nomes 1:1).** Atual (03/07): **154 tabelas** (47 CONNECTOR + 107 GCS_FARM), **23 views** (6 CONNECTOR + 17 GCS_FARM), 1 procedure.

> **Fase B (05/07): espelho tipado `FARMBOX_*` eliminado do `GCS_FARM`.** As 29 tabelas `FARMBOX_*` + 4 views `VW_FARMBOX_*` foram dropadas do master. O `CONNECTOR_GCS_FARM` (landing) permanece recebendo o JSON cru da API Farmbox (tabelas `FARMBOX_*` com coluna `record` JSON). O ETL agora lê o JSON cru **direto do CONNECTOR** (`JSON_VALUE`/`OPENJSON` sobre `record`) e grava **direto no domínio nativo `FARM_*`** — não há mais passo de espelho tipado dentro do `GCS_FARM`. As contagens acima (03/07) ainda refletem o estado pré-Fase B; revalidar após o próximo `SETUP_FULL.sql`.

## Arquivos
| Arquivo | O que é |
|---|---|
| `SETUP_FULL.sql` | **Núcleo canônico.** Build dos dois bancos (CONNECTOR → GCS_FARM): núcleo + FARM_* base + FERT_* + OPS_* + VRA. Re-executável. |
| `RESET_FULL.sql` | Dropa `CONNECTOR_GCS_FARM` e `GCS_FARM` inteiros (só local/teste; em produção, **nunca**). |
| `MODULE_AGRO_V1.sql` | **Módulo agronômico nativo** (FARM_* produtos/bulário/aplicações/monitoramento/pragas/contagem/estimativa). Roda **depois** do SETUP_FULL. |
| `FLIGHT_LOG.sql` | **Aplicação Aérea** (FLIGHT_LOG*). Roda depois do MODULE_AGRO (FK → FARM_APPLICATION). |
| `MONITOR_STOP_RESULTS.sql` | **Achados POR PARADA (ponto) + limites de controle** — cria `FARM_PEST_THRESHOLD` (base) + `FARM_MONITORING_STOP_RESULT` (achado por ponto: lat/lon + praga + quantidade) e **materializa do raw** (`FARMBOX_MONITORING.record` → `monitoring_stops[].monitoring_stop_results[]`, ~842k linhas). Fonte do mapa de calor. Roda depois do MODULE_AGRO. |
| `MONITOR_THRESHOLDS_CSV.sql` | **Limites de controle POR CULTURA (seed do CSV Farmbox).** Adiciona `FARM_PEST.category` + `FARM_PEST_THRESHOLD.rep_action_level/rep_damage_level/max_per_sample` e semeia por (cultura, alvo, estágio) com Controle/Dano Vegetativo+Reprodutivo (740 linhas, 9 culturas, 4 categorias). ⚠️ **aplicar com `sqlcmd -f 65001 -I` (UTF-8)** — senão nomes acentuados corrompem e perde-se metade das culturas. Roda depois do MONITOR_STOP_RESULTS. |
| `MONITOR_NOTES.sql` | **Notas georreferenciadas com fotos** — cria `FARM_MONITORING_NOTE` (lat/lon + descrição + `image_addresses[]` S3) e materializa do `FARMBOX_MONITORING_DAY_RESULT.record.notes[]` (~57k, 47k com foto). Fonte do clique-no-ponto/camada de fotos do mapa de calor. |
| `MONITOR_CARENCIA.sql` | **Carência de insumos** — `FARM_PRODUCT_CARENCIA_DEFAULT` (default por categoria de produto: reentrada+colheita) + `FARM_PRODUCT_CARENCIA` (override por produto/bulário) + seed conservador. Roda **antes** do MODULE_MONITOR_V1 (a view usa essas tabelas). |
| `MODULE_MONITOR_V1.sql` | **Monitoramentos — config que comanda o app** (MONITOR_TOLERANCE_DEFAULT [default global por cultura] + MONITOR_TOLERANCE [exceções] + METHODOLOGY/FIXED_POINT/REQUEST + estende FARM_MONITORING + `VW_MONITOR_FIELD_STATUS`, cuja carência usa a REENTRADA resolvida override-produto›default-categoria). Roda depois do MODULE_AGRO + MONITOR_CARENCIA; idempotente. |
| `MODULE_RESEARCH_V1.sql` | **Módulo Pesquisa — ensaios de faixa (strip test / lado a lado).** Cria `FARM_RESEARCH_TRIAL` (1 ensaio por talhão+ciclo) + `FARM_RESEARCH_STRIP` (N faixas: variedade + polígono GEOGRAPHY + área + produtividade + `farmbox_plantation_id`) com índice espacial `SIX_FARM_RESEARCH_STRIP`. **Aditivo** — não toca FARM_FIELD_PLANTING. Backfill do histórico via `research.service.backfillResearchFromFarmbox()` (agrupa plantações por talhão+ciclo com ≥2 variedades). Roda depois do MODULE_AGRO + planejamento (FK → FARM_FIELDS/FARM_SEASON_CYCLE/FARM_VARIETY). Idempotente. |
| `MODULE_PHENOLOGY_V1.sql` | **Fenologia — catálogo de estágios por cultura + GUIA DE CAMPO.** Cria `FARM_PHENOLOGICAL_STAGE` (por cultura: `code`/nome, `position`, `classification` veg/rep, `description`, `ignore_infestations`, `farmbox_stage_id` + **guia curado**: `id_tips` [dicas de identificação], `days_after_emergence_min/max`, `confused_with_ids`) + `FARM_MONITORING.phenological_stage_id` (FK). Materializado do `FARMBOX_REF_PHENOLOGICAL_STAGE`; a string do monitoramento é normalizada + resolvida à FK (`phenology.service`, SQL compartilhado com o ETL). **Guia de campo** (curadoria do agrônomo, o ETL NÃO toca): `FARM_MEDIA` (blob genérico servível, dedup por sha256) + `FARM_PHENOLOGICAL_STAGE_MEDIA` (N fotos/vídeos por estágio, upload OU URL externa). Aditivo/idempotente. Roda depois de MODULE_AGRO + planejamento. |
| `WEATHER_SEASON_SUMMARY.sql` | **Meteorologia — resumo de clima por safra (rollup por plantio).** Cria `WEATHER_SEASON_SUMMARY` (1 linha por `FARM_FIELD_PLANTING`: janela + precip/temperaturas/umidade/radiação/dias-de-chuva + irrigação + `source` auto/manual). Precip/temp/umidade/radiação são AGREGADOS da `FIELD_WEATHER_HOURLY` na janela do plantio (via `weatherSeason.service.recomputeSeasonClimate`, que NÃO sobrescreve linhas `source='manual'`); irrigação é manual. Aditivo. Roda depois de MODULE_AGRO + planejamento + o grid de clima. Idempotente. |
| `FERT_EXPORT_PROFILES.sql` / `FERT_CROP_EXPORT_SCOPE.sql` | Perfis/escopo de exportação de fertilidade. |
| `MATERIALIZE_FARM.sql` / `DROP_FARMBOX_MIRROR.sql` | Materialização FARM_* a partir do JSON cru / drop do espelho Farmbox (Fase B). Espelho SQL do `farmMaterialize.service.ts` (fonte-da-verdade é o serviço TS, que também dobra stop-results/notas/thresholds/carência). |

> **Ordem de execução:** `SETUP_FULL` → `MODULE_AGRO_V1` → `FLIGHT_LOG` → `MONITOR_STOP_RESULTS` → `MONITOR_THRESHOLDS_CSV` **(`sqlcmd -f 65001 -I`)** → `MONITOR_NOTES` → `MONITOR_CARENCIA` → `MODULE_MONITOR_V1` → `MODULE_RESEARCH_V1` → `MODULE_PHENOLOGY_V1` → `WEATHER_SEASON_SUMMARY` → `FERT_*`. A DDL COMPLETA do domínio é a soma desses (módulos com FK para FARM_APPLICATION não cabem no SETUP_FULL). Ver `../AUDITORIA_2026-07-07.md`.

> **Fonte única:** o `SETUP_FULL.sql` é o **único** SQL de schema. Os recortes por módulo (`modulos/`) e as cópias legadas em `Arquivos Suporte/` foram **removidos (03/07)** — não há mais duplicata que possa dar drift.

## Como usar
- **Build do zero (dev/teste):** `RESET_FULL.sql` → `SETUP_FULL.sql`.
- **Produção (1ª vez):** rodar só o `SETUP_FULL.sql` (não dropa nada; cria o que falta).
- **Evolução / novo módulo:** edite **direto o `SETUP_FULL.sql`** (canônico) e aplique no banco. Sem recortes por módulo — tudo vive num arquivo só.

## Regra de ouro
**O `SETUP_FULL.sql` é a única fonte da verdade e tem que casar 1:1 com o banco.** Nada de objeto criado "só no banco" sem refletir aqui, e nenhuma cópia paralela do SQL. Ver também `../04_Guia_Arquitetura_e_Novos_Modulos.md`.

> Núcleo compartilhado (não é "módulo", vive no SETUP_FULL): CONFIG (inclui `CONFIG_SCHEDULER` + `CONFIG_SCHEDULER_LOG` — agendador central), MANAGEMENT (inclui `MANAGEMENT_USER_PREFERENCE`), FARM_* (fazenda/gleba/talhão/geometria + **calendário** `FARM_SEASON`/`_CYCLE`) e VRA.
>
> **Planejamento agrícola, cultivos & produtividade** (tudo no SETUP_FULL): `FARM_VARIETY`/`FARM_VARIETY_TRAIT`/`FARM_VARIETY_TRAIT_VALUE` (variedades + características configuráveis), `FARM_FIELD_PLANTING` (plantio/produtividade real), `FARM_FIELD_ROTATION` + `FARM_PLOT_ROTATION`/`_CROP` (rotação), `FARM_PLANTING_REVIEW` (outliers), `PROD_ESTIMATE_FORMULA` (fórmula da estimativa). Fertilidade ganhou `FERT_CROP_EXPORT`/`FERT_EXPORT_NUTRIENT` (exportação de nutrientes) e `FERT_AMENDMENT_APPLICATION` (corretivos). O amostrador da contagem é lido direto do JSON cru do `CONNECTOR_GCS_FARM` (`..._MONITORING_DAY_RESULT.record.monitors[]`). A DDL desses já está no `SETUP_FULL.sql`; **seeds** de `PROD_ESTIMATE_FORMULA`/`FERT_CROP_EXPORT` são **valores iniciais** (a formula/coeficiente pode ser editada no app e diverge do seed sem ser "drift" de schema).
